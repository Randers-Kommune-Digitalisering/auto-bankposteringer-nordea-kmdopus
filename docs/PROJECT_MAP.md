# Project map (quick mental model)

This document is a **navigation aid**: what lives where, how runtime flows through the system, and which files are the “entry points”.

Docs index: `docs/README.md`

## What this repo is

A stateless financial integration engine + UI:

- All domain state lives in Postgres.
- Runtime behavior is deterministic from persisted state + inputs.
- Background work is modeled as jobs + outbox.

## Top-level areas

### app/
Nuxt UI + shared app utilities.

Key files:

- Environment validation: `app/lib/env.ts` (full runtime env) and `app/lib/env-database.ts` (DB-only env for migrations/seed).
- Database client + schema: `app/lib/db/index.ts` and `app/lib/db/schema/*`.

### server/
Nitro server (API routes, tasks, plugins).

Key files:

- Scheduled tasks: `server/tasks/*`.
- Always-on background worker loop: `server/plugins/worker-loop.server.ts`.
- Role gating helpers: `server/utils/appRole.ts`.
- Health endpoint: `server/api/health.get.ts`.

### engine/
The domain engine (handlers + infrastructure + ports).

Key concepts:

- Queue worker implementation: `engine/queue/handlers/worker.ts`.
- Job enqueue: `engine/queue/handlers/enqueueJob.ts`.
- Banking ingestion: `engine/banking-ingestion/**`.
- ERP integration: `engine/erp-integration/**`.

### scripts/
Run-once scripts (seed/reset).

- System seed: `scripts/db/seed-system.ts`.

### drizzle/
Database migrations generated by Drizzle.

- Migration runner config: `drizzle.config.ts`.

### resources/
Static resources and fixtures used by the engine/tests (e.g. bundled example bank documents).

## Runtime roles (production)

The same container image is deployed in different roles via `APP_ROLE`:

- `web`: UI + HTTP API. Scales horizontally.
- `scheduler`: runs scheduled tasks (cron-like). Should be 1 replica. Enqueue-only.
- `worker`: processes jobs/outbox continuously. Can scale independently.

See `docs/ARCHITECTURE.md` for the operator view.

## Main flow: scheduler → job → worker

1) **Scheduler** triggers the scheduled task `bank-transactions-batch`.
2) The task **enqueues** a `job` row (e.g. `banking.ingest`). It uses a Postgres advisory lock so only one scheduler instance enqueues per tick.
3) The **worker loop** continuously runs `runWorker()`.
4) `runWorker()` claims jobs (`FOR UPDATE SKIP LOCKED`) and executes `handleJob(type, payload)`.
5) Some actions create outbox items; the worker also claims and processes outbox.

## IO vs CPU isolation

Workers can be started with `WORKER_PROFILE`:

- `all` (default): jobs + outbox
- `cpu`: CPU/DB heavy jobs only (e.g. `banking.ingest`)
- `io`: outbox + IO heavy jobs (e.g. `erp.ingestResponses`)

Optional tuning env vars are documented in `docs/ARCHITECTURE.md`.
